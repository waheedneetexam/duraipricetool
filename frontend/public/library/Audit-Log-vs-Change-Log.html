Audit Log vs Change Log

Purpose: Document why two distinct trails are needed, how they differ, and how tenant isolation is enforced for each.

Why we need audit logs

  • Proof of control: Audit logs give regulators and security teams a cryptographic trail showing who did what and when.
  • Tenant isolation: APIs always filter on actor_tenant_id when a tenant-level user is querying logs, so Region A cannot read Region B’s investigations.
  • Security callbacks: When something abnormal happens (SKU tampering, workflow overrides), the audit log tells you who touched the system and with what permission.
  • Rollbacks and accountability: Audit entries help you identify the source of misconfigurations before chasing rollbacks.

Why we need change logs

  • Operational visibility: Business users want a friendly record of data changes (quotes, master data, formulas) without reading technical audit data.
  • Tenant trust: Each client sees a change log scoped to their records, reinforcing that we only touch their data.
  • Workflow context: Change logs tie directly to specific domain objects, making reviews easier.
  • Approval support: Before applying a configuration change, reviewers read the change log to understand the before/after state.

Key differences

Dimension Audit Log Change Log
Focus Security, compliance, permissions, actor context. Business context, domain data changes, before/after values.
Granularity Row action metadata with actor IDs and timestamps. Field-level dataset changes scoped to specific objects.
Audience Security, compliance, platform ops, SuperAdmin. Operational teams, Tenant admins, change approvers.
Retention Retention aligns with governance/regulatory policies. Retention matches tenant-specific operational needs.
Tenant filtering Always filtered by actor_tenant_id; only SuperAdmin with platform.audit.read sees across tenants. Scoped to the tenant’s own records, so each region reviews their updates only.

Tenant-specific implementation notes

  • Tenant audit access: The `/audit/admin` endpoint passes the caller’s tenant ID to `list_audit_logs`, building a SQL clause like actor_tenant_id = ? to guarantee separation.
  • SuperAdmin visibility: SuperAdmin users holding `platform.audit.read` may call `/audit/platform` to span tenants for governance reviews.
  • Change log isolation: Change logs attach to tenant-tagged records (quotes, master data, workflows), so the UI only surfaces the tenant’s own history.
Keep both trails: Audit logs satisfy governance, while change logs satisfy tenant transparency. Together they deliver the accountability and traceability that multi-region pharma clients expect.
