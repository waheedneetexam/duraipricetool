Phase 1 — Backend: User & Tenant Management API IN PROGRESS

New file: app/api/routes_platform.py — SuperAdmin routes

Requires platform.tenants.manage or platform.users.manage permission.

MethodRouteDescription
GET/platform/tenantsList all tenants
POST/platform/tenantsCreate a new tenant
PATCH/platform/tenants/{id}Activate / suspend tenant
GET/platform/usersList all users across tenants
POST/platform/usersCreate platform user
POST/platform/users/{id}/assign-tenantAssign user to tenant with role

New endpoints in app/api/routes_admin.py — TenantAdmin routes

Requires tenant.users.manage or tenant.roles.assign.

MethodRouteDescription
GET/admin/usersList users in current tenant
POST/admin/usersCreate user inside current tenant
POST/admin/users/{id}/assign-roleAssign business role (blocks SuperAdmin)
DELETE/admin/users/{id}/role/{role}Remove role from user in tenant

New file: app/services/platform_service.py

All DB logic for user/tenant CRUD, scoped to DuckDB (and Postgres in hybrid mode).

Phase 2 — Backend: Audit Log

New audit_log table, audit_service.py, and read endpoints. All SuperAdmin cross-tenant actions are logged.

Phase 3 — Frontend: New Admin Tabs

  • Users tab (shown for tenant.users.manage): manage users within the current tenant
  • Platform tab (shown for platform.tenants.manage): SuperAdmin tenant + cross-tenant user management

Phase 4 — Hard Restrictions

  1. TenantAdmin cannot read or modify data belonging to another tenant (enforced by tenant_id scoping in every query).
  2. TenantAdmin cannot assign or remove the SuperAdmin role (enforced in the assign-role endpoint).
  3. All SuperAdmin cross-tenant operations are written to the audit log.

Deferred (Out of Scope)

  • Billing / Licensing — no billing provider integrated yet
  • SSO / Password policy enforcement — complex, deferred
  • Email invitations — manual password set for now
