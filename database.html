6) Classification Catalog

Admin-controlled table that records how each table is categorized.

TableNotes
table_classificationsStores `table_name`, `category`, and `tenant_id`. Managed via Data Management Admin and ensures the dropdown reflects the latest classification.

The combination of `table_name` and `tenant_id` is the primary key. Each row persists the chosen bucket from master_data, transactional_data, configuration_data, metadata, or organization_data. Timestamps (`created_at`, `updated_at`) capture when a category was last refreshed.

To update a classification, use the Data Management Admin UI or call the admin endpoints directly:

  • GET /admin/data/table-classifications (permission master_data.read) returns the active mapping for the tenant.
  • PUT /admin/data/table-classifications/{table_name} (permission master_data.manage) saves a new bucket.
  • The frontend dropdown reads app/constants/table_classification.ts to render the same five buckets and keeps the display in sync with the database table.

Every new table created for a tenant should have its classification saved here and the corresponding category documented in this file.

How to Distinguish the Categories

  • Master Low‑change reference data. Usually referenced by transactions. Primary IDs are stable.
  • Transactional Time‑series records representing events. High volume. Usually has created_at/updated_at.
  • Configuration Rules/settings that change system behavior. Edited by admins.
  • Metadata System‑level operational info (sync/audit).
  • Organization Tenancy, users, roles, and permissions.

Non‑Breaking Code Changes (Recommended)

Avoid moving tables or renaming columns. Add classification in code and docs only.
  • Add a static classification map (documentation only). Example file: app/data/data_classification.py.
  • If you want to use it in the UI later, load it in a read‑only endpoint without altering existing endpoints.
  • Prefer additive changes: new columns, new tables, or new views. Avoid altering existing queries.
  • Use a feature flag for any new behavior (e.g., ENABLE_DATA_CATALOG).
  • Keep all current API contracts unchanged while adding new ones.
