Does P-Competitor implement multi-tenant login and roles similar to our plan?

Checked on: March 5, 2026

Short Answer

Yes, partially similar. P-Competitor supports authentication, role assignment, and permission-based access. For tenant isolation, P-Competitor uses a partition-centric model (tenant-like separation by partition), which is different from a typical single-app row-level tenant_id model.

What is Similar vs Different

Capability P-Competitor Compared to your DuraiPricingTool plan
Login/Auth Supports Basic, JWT, OAuth2 (auth code + PKCE), SAML, external JWT trust. Similar and enterprise-grade.
Roles Supports user roles and role assignment APIs. Similar RBAC capability.
Permissions Permission checks exist; API trust can further restrict allowed permissions. Similar concept to permission keys and route guards.
Multi-tenancy shape Primarily partition-based isolation (credentials/token carry partition context). Different from one database with row-level tenant_id for all customers.
Single user across many tenants in one login context Not the same model as in-app tenant switching inside one shared dataset; usually partition-context access. Your plan supports tenant switcher and tenant-scoped roles in one app.

If Not Exactly Similar, How to Align

  1. Keep your current single-platform model with tenant_id on all business tables.
  2. Implement tenant-scoped RBAC in user_tenant_roles.
  3. Issue JWT with tenant_id + permission claims.
  4. Enforce tenant filter in every query and API endpoint.
  5. Add tenant switch API/UI and audit logs for every role-sensitive action.

Recommendation for DuraiPricingTool

Use P-Competitor as a benchmark for enterprise auth patterns (JWT/OAuth2/SAML and strong RBAC), but implement multi-tenant behavior in DuraiPricingTool using explicit tenant_id isolation and tenant-scoped roles because your pharma use case needs a single app serving multiple regional clients with fine-grained control.
